# Building

Requirements:
- Rust toolchain (install via rustup)

Nice-to-have:
- [`just` task runner](https://just.systems/)
- [`cargo-insta` testing tool](http://insta.rs/docs/quickstart/)

Check the [`justfile`](./justfile) for commands to run common tasks.

# Making releases

Releases go through a pull request. The version-bump PR is the source of
truth: once it lands on `main`, the rest of the pipeline takes over.

## 1. Open the release PR

For version `X.Y.Z`:

1. Cut a branch: `git checkout -b release/X.Y.Z`
2. Edit [`CHANGELOG.md`](./CHANGELOG.md): rename the `# Unreleased` header to
   `# X.Y.Z` and add a fresh empty `# Unreleased` section above it.
3. Bump `version` in [`Cargo.toml`](./Cargo.toml).
4. Run `cargo build` so `Cargo.lock` picks up the new version.
5. Commit as `release: X.Y.Z` and push.
6. Open a PR titled `release: X.Y.Z`. Review focuses on the changelog wording.

## 2. Merge

When the PR lands on `main`, the [`tag-release`](./.github/workflows/tag-release.yml)
workflow detects the `Cargo.toml` version bump and pushes a `vX.Y.Z` tag.

The tag push triggers [`release.yml`](./.github/workflows/release.yml)
(generated by [`cargo-dist`](https://github.com/axodotdev/cargo-dist), config in
[`dist-workspace.toml`](./dist-workspace.toml)) which builds the binaries and
publishes a GitHub Release.

## One-time setup: `RELEASE_TAG_PAT`

For the tag pushed by `tag-release.yml` to trigger `release.yml`, it must be
pushed using a Personal Access Token, not the default `GITHUB_TOKEN` (GitHub
suppresses workflow runs caused by the default token to prevent loops).

Configure a repo secret `RELEASE_TAG_PAT` (fine-grained PAT with `contents:
write` on this repo). Without it, `tag-release.yml` will detect the bump but
skip pushing the tag and print the exact command to run manually.

## Manual fallback

If the workflow is unavailable (or the PAT isn't configured), tag from your
machine after the release PR merges:

```sh
git fetch origin main
git tag vX.Y.Z origin/main
git push origin vX.Y.Z
```
