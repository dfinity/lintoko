# Making releases

Releases go through a pull request. Once the PR is merged, automation pushes
the tag and `cargo-dist` publishes the GitHub Release.

## 1. Open the release PR

For version `X.Y.Z`:

1. Cut a branch: `git checkout -b release/X.Y.Z`
2. Edit [`CHANGELOG.md`](./CHANGELOG.md): rename the `# Unreleased` header to
   `# X.Y.Z` and add a fresh empty `# Unreleased` section above it.
3. Bump `version` in [`Cargo.toml`](./Cargo.toml).
4. Run `cargo build` so `Cargo.lock` picks up the new version.
5. Commit as `release: X.Y.Z` and push.
6. Open a PR with title **exactly** `release: X.Y.Z` and apply the **`release`** label.

## 2. Wait for `release-pr` validation

The [`release-pr`](./.github/workflows/release-pr.yml) workflow runs on every
push to a labeled release PR and checks that:

- the PR title is `release: X.Y.Z`
- `Cargo.toml`'s version equals `X.Y.Z`
- `CHANGELOG.md` has a `# X.Y.Z` section

If any of these fail, fix them in the PR before merging.

## 3. Merge

On merge, `release-pr` pushes a `vX.Y.Z` tag pointing at the merge commit. The
tag triggers [`release.yml`](./.github/workflows/release.yml) (generated by
[`cargo-dist`](https://github.com/axodotdev/cargo-dist), config in
[`dist-workspace.toml`](./dist-workspace.toml)) which builds the binaries and
publishes the GitHub Release.

## Infrastructure notes

`release-pr` pushes the tag using a GitHub App token via
[`actions/create-github-app-token`](https://github.com/actions/create-github-app-token).
This is required because tags pushed by the default `GITHUB_TOKEN` do not
trigger downstream workflows (so `release.yml` would never fire). The App
credentials come from org-level config and are referenced as
`vars.GENERIC_CI_RW_APP_ID` / `secrets.GENERIC_CI_RW_APP_PRIVATE_KEY` -- the
same App used by `dfinity/motoko` and the mops CLI release workflow.

If the org App is unavailable (workflow fails on the `Create GitHub App Token`
step), tag manually after merging:

```sh
git fetch origin main
git tag vX.Y.Z origin/main
git push origin vX.Y.Z
```
